RENDU — PROJET DDD — GESTION DE DEVIS ET DE FACTURATION

Ce document décrit comment modéliser et structurer un logiciel pour la gestion du cycle devis → facturation → paiements, dans une démarche DDD (Domain-Driven Design). Il est pensé pour qu’un nouveau lecteur comprenne d’emblée le cadre avant de parcourir les listes d’événements, de scénarios et de contextes.

1. BDD (GHERKIN)

Explications
Le Behavior-Driven Development (BDD) consiste à décrire le comportement attendu du système dans un langage lisible par le métier et les développeurs. Le format Gherkin (Feature, Scenario, Given / When / Then / And) structure ces exemples concrets.

Ci-dessous : scénarios exprimant les comportements essentiels du logiciel de devis et facturation.
Tous les scénarios utilisent volontairement le format Gherkin strict Given / When / Then / And, avec des exemples concrets.

  En tant qu'équipe financière
  Je veux suivre précisément les statuts d'une facture
  Afin de savoir si elle est émise, réalisée, payée, annulée ou clôturée

  Scenario: Passage d'une facture émise à réalisée
    Given une facture "Facture Devis audit DDD - Martin" au statut "Émise"
    And la prestation "Audit DDD" est planifiée pour la cliente "Lucie Martin"

    And le changement de statut est ajouté à l'historique

  Scenario: Passage d'une facture réalisée à soldée

    When un paiement total de 500€ est enregistré par virement bancaire
    Then la facture passe au statut "Soldée"

  Scenario: Clôture automatique d'une facture réalisée et soldée

2. RÉPARTITION DES TÂCHES

Note : le document source ne contenait pas de répartition figée ; proposition alignée sur les bounded contexts identifiés.

Zones / équipes et tâches types
- Gestion des devis : modèle agrégat devis, versionnement, lignes, notes, statuts, API et UI création/édition.
- Gestion de la facturation : génération facture depuis devis approuvé, titrage, échéance, historique de statuts, immuabilité documentaire.
- Paiements et relances : enregistrement paiements, reste à charge, détection retard, relances, intégration modes de paiement.
- Gestion des avoirs : cycle de vie avoir, compensation / remboursement, annulation, expiration.
- Import / Export : import Excel devis/factures, export CSV, Excel, PDF ; fabrique de devis depuis import.
- Notification et envoi : envoi e-mail devis/factures, gabarits, file d’événements.
- Traçabilité et preuve : historique versions, audit, éventuelle intégration blockchain (hors périmètre fonctionnel minimal).

Ordre de dépendance suggéré : Devis → Facturation → (Paiements + Notifications) en parallèle partielle ; Import/Export en transversal ; Traçabilité en transversal dès les premières itérations.

3. DÉFINIR LES BOUNDED CONTEXTS

Explications (bounded context)
Un bounded context (« contexte délimité ») est une zone du logiciel où le vocabulaire et les règles métier veulent dire une seule chose, sans ambiguïté. Chaque contexte est comme un « quartier » : tant qu’on reste dedans, tout le monde est d’accord sur le sens des mots et sur les règles.

Points clés
- Même mot, sens différent selon le contexte (ex. « Facture » strictement comptable en Facturation vs résumé utile en Paiements & relances).
- Frontière = protection contre la confusion : évite un modèle géant contradictoire.
- Organisation du code et des équipes : correspond souvent à un module (ou microservice) et parfois à une équipe.
- Communication contrôlée : via des contrats explicites (événements, API, messages). Si besoin, traduction via anti-corruption layer.

Carte de contexte (définition)
La carte de contexte est une vue d’ensemble (souvent graphique) montrant les bounded contexts du système, leurs relations (qui consomme quoi, amont/aval) et parfois les patterns d’intégration (partenariat, client–fournisseur, conformiste, anti-corruption layer…). Elle sert à comprendre les dépendances entre parties du logiciel et entre équipes, et à décider où placer une fonctionnalité.

3.1. BOUNDED CONTEXTS IDENTIFIÉS (SYNTHÈSE)

Pour chaque contexte : type DDD, responsabilité principale, ce qu’il ne doit pas gérer.

- Négociation & Vente / Gestion des devis
  - Type DDD : Core Domain
  - Responsabilité : créer, modifier, versionner, envoyer et faire approuver les devis.
  - Ne doit pas gérer : paiements réels, relances, règles comptables de facture.

- Facturation
  - Type DDD : Supporting Domain
  - Responsabilité : générer une facture depuis un devis approuvé, gérer l’échéance, les statuts comptables, corrections par avenant.
  - Ne doit pas gérer : négociation commerciale, encaissement bancaire.

- Trésorerie & Recouvrement / Paiements et relances
  - Type DDD : Supporting Domain
  - Responsabilité : suivre les paiements, calculer le reste à charge, détecter les retards, déclencher les relances.
  - Ne doit pas gérer : création détaillée du devis, mise en forme finale de la facture.

- Gestion des avoirs
  - Type DDD : Supporting Domain
  - Responsabilité : gérer les avoirs issus d’une correction, d’une annulation ou d’un remboursement partiel.
  - Ne doit pas gérer : contenu initial d’une facture (réagit à un avenant ou à une demande financière).

- Catalogue & Tiers
  - Type DDD : Shared Kernel
  - Responsabilité : fournir les clients, produits et services utilisés dans devis et factures.
  - Ne doit pas gérer : cycle de vie d’un devis, d’une facture, d’un paiement.

- Import / Export documentaire
  - Type DDD : Generic Domain
  - Responsabilité : importer (Excel), exporter (CSV, Excel, PDF), transformer des données externes en objets métier valides.
  - Ne doit pas gérer : contournement des règles métier (les validations s’appliquent aussi à l’import).

- Notification et envoi
  - Type DDD : Infrastructure / Supporting Domain
  - Responsabilité : envoyer devis/factures/relances par e-mail, gérer modèles et suivi d’envoi.
  - Ne doit pas gérer : décider si un devis est approuvé ou si une facture est payée.

- Traçabilité et preuve
  - Type DDD : Infrastructure / Supporting Domain
  - Responsabilité : historique des versions, statuts, actions importantes ; préparation d’une preuve d’authenticité (ex. blockchain).
  - Ne doit pas gérer : modifier les règles métier (observe et enregistre les faits).

3.2. DESCRIPTION DES INTERACTIONS

Parcours principal : devis → facture → paiement → clôture
1) Négociation & Vente crée le devis (client, produits/services, montants, notes, conditions).
2) À chaque modification importante : nouvelle version + événement NouvelleVersionDevisCreee.
3) À l’envoi : Notification et envoi transmet au client + trace l’envoi.
4) À l’approbation : événement DevisApprouve.
5) Facturation consomme DevisApprouve et génère la facture (titre, lignes, taxes, client, dates, échéance).
6) Facturation publie FactureEmise ; Notification et envoi envoie la facture.
7) À la réalisation : Facturation passe à Réalisée et publie FactureRealisee.
8) Trésorerie & Recouvrement enregistre les paiements et publie PaiementEnregistre / FacturePartiellementPayee.
9) Si reste à charge = 0 : Facturation passe à Soldée puis (si réalisée) à Clôturée.
10) Si échéance dépassée : événement EcheanceDepassee ; Notification et envoi transmet une relance.
11) En cas de correction : Facturation publie AvenantValide.
12) Gestion des avoirs consomme AvenantValide et crée un avoir (remboursement AvoirRembourse ou compensation AvoirCompense).
13) Import / Export interagit avec Négociation & Vente et Facturation sans contourner les validations.
14) Traçabilité et preuve écoute les événements importants (création, modification, approbation, émission, paiement, relance, avoir, export).

Règles de lecture formelles
- Upstream / Downstream : Négociation & Vente est upstream de Facturation ; Facturation est upstream de Trésorerie & Recouvrement et Gestion des Avoirs.
- Shared Kernel : Catalogue & Tiers est partagé entre Négociation & Vente et Facturation.
- ACL : Import / Export Documentaire passe par une couche d’anticorruption pour protéger les modèles métiers internes.
- Conformist : Trésorerie & Recouvrement se conforme au modèle de facture publié par Facturation pour éviter la duplication des règles comptables.
- Traçabilité transversale : tous les contextes émettent des traces ou événements vers Traçabilité & Preuve.

3.3. LANGAGE UBIQUITAIRE — CORE DOMAIN — INFRASTRUCTURE

1) Core Domain — Négociation & vente
- Dossier commercial / affaire : agrégat regroupant plusieurs devis, factures ou avoirs liés à un même besoin client.
- Devis : agrégat racine (proposition chiffrée) contenant client, lignes, notes, conditions, statut, versions.
- Version de devis : instantané ; une modification importante crée une nouvelle version.
- Ligne de devis : produit/service, quantité, prix unitaire, remise éventuelle, taxes.
- Statut du devis : Brouillon, Envoyé, En négociation, Approuvé, Refusé, Expiré.
- Notes : texte libre client ou interne.
- Conditions spécifiques : clauses commerciales/légales.
- Règles :
  - Titre + client + au moins une ligne (montant ≥ 0) obligatoires.
  - Date de création automatique et immuable.
  - Modification autorisée seulement en Brouillon ou En négociation.
  - Modification importante ⇒ nouvelle version ; seule la dernière version peut être approuvée.
  - Devis approuvé immuable ; déclenche facturation.
- Événements publiés : DevisCree, DevisModifie, NouvelleVersionDevisCreee, DevisEnvoye, DevisApprouve, DevisRefuse, DevisExpire.

2) Supporting Domain — Facturation
- Facture : agrégat racine (document comptable officiel).
- Ligne de facture : dérivée de ligne de devis, figée à l’émission.
- Date d’échéance : date limite de paiement.
- Statut de facture : Émise, Réalisée, Partiellement soldée, Soldée, Annulée, Clôturée.
- Historique de statuts : suivi des changements.
- Avenant : correction d’une facture émise.
- Profil de facturation : logo, SIRET, mentions légales, IBAN.
- Règles :
  - Facture générée automatiquement depuis devis approuvé.
  - Titre : « Facture » + titre du devis.
  - Facture émise immuable ; correction via avenant.
  - Chaque statut apparaît au plus une fois dans l’historique.
  - Facture Soldée et (a été) Réalisée ⇒ Clôturée automatiquement.
- Événements publiés : FactureEmise, FactureEnvoyee, FactureRealisee, FactureAnnulee, FactureCloturee, AvenantGenere, AvenantValide.

3) Supporting Domain — Trésorerie & recouvrement
- Paiement : transaction partielle ou totale.
- Mode de paiement : virement, carte, chèque, etc.
- Reste à charge : montant restant après paiements et compensations.
- Échéancier : plan de paiement.
- Relance : rappel après échéance dépassée.
- Mise en demeure : escalade après relances.
- Règles :
  - Paiement partiel : diminue le reste à charge, n’arrête pas la surveillance.
  - Paiement total : reste à charge = 0, permet soldée.
  - Facture non soldée après échéance : déclenche relance.
  - Mise en demeure : peut informer la vente pour bloquer de nouveaux devis.
- Événements publiés : PaiementEnregistre, PaiementRejete, PaiementRembourse, FacturePartiellementPayee, EcheanceDepassee, RappelPaiementEnvoye, MiseEnDemeureEmise.

4) Supporting Domain — Gestion des avoirs
- Avoir : montant dû au client, utilisable par remboursement ou compensation.
- Remboursement : sortie d’argent.
- Compensation : diminution du reste à charge d’une autre facture.
- Statuts d’avoir : Disponible, Compensé, Remboursé, Expiré, Annulé.
- Règles :
  - Avoir possible après avenant validé / correction à la baisse.
  - Pas de double consommation (pas remboursé et compensé pour le même montant).
  - Avoir remboursé ou compensé doit être clôturé.
- Événements publiés : AvoirCree, AvoirApplique, AvoirRembourse, AvoirExpire, AvoirAnnule, AvoirCompense.

5) Shared Kernel — Catalogue & tiers
- Client : destinataire des devis/factures.
- Produit / service : élément du catalogue.
- Devise : monnaie des montants.
- Taxe / TVA : taux de taxe.
- Règles :
  - Référencé par les autres contextes sans ajouter de règles contradictoires.
  - Doit rester simple et stable pour éviter le couplage.

6) Generic / Supporting Domain — Import / export documentaire
- Import Excel : transformation de données externes en devis/facture valide.
- Export CSV / Excel / PDF : génération documentaire.
- Fabrique de devis depuis import : service de transformation + validation.
- Règles :
  - Import = mêmes règles que création manuelle.
  - Export = données validées par le contexte propriétaire.
  - Import invalide : signaler les erreurs plutôt que créer un objet incomplet.

7) Infrastructure / Supporting Domain — Notification et envoi
- E-mail de devis, e-mail de facture, relance de paiement, gabarits.
- Règles :
  - Ne décide pas des statuts ; exécute des demandes d’envoi.
  - Chaque envoi important est tracé.

8) Infrastructure / Supporting Domain — Traçabilité et preuve
- Historique d’action, historique de version, journal d’événements, preuve d’authenticité (option blockchain).
- Règles :
  - Observe les faits ; ne modifie pas les décisions métier.
  - Événements critiques horodatés et rattachés au document.
  - Blockchain : option envisagée, pas obligatoire au périmètre minimal.

ÉVÉNEMENTS DU DOMAINE STRUCTURANTS
- DevisCree : un devis valide a été créé.
- NouvelleVersionDevisCreee : une modification importante a généré une nouvelle version.
- DevisEnvoye : le devis a été transmis au client.
- DevisApprouve : le client a accepté le devis ; déclenche la facturation.
- DevisExpire : le devis ne peut plus être accepté.
- FactureEmise : une facture a été générée à partir d’un devis approuvé.
- FactureEnvoyee : la facture a été transmise au client.
- FactureRealisee : le produit ou service associé a été réalisé.
- PaiementEnregistre : un paiement a été reçu et doit mettre à jour le reste à charge.
- FacturePartiellementPayee : la facture n’est pas encore totalement soldée.
- EcheanceDepassee : la date limite de paiement est dépassée.
- RappelPaiementEnvoye : une relance a été envoyée.
- MiseEnDemeureEmise : une relance forte a été déclenchée.
- AvenantValide : une correction de facture validée peut créer un avoir.
- AvoirCree : un avoir est disponible.
- AvoirRembourse : l’argent est effectivement reparti vers le client.
- AvoirCompense : un avoir a diminué la dette d’une autre facture.
